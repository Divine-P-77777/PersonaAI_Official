## Part 2: System Architecture

**(Visual: High-level Architecture Diagram showing Next.js -> FastAPI -> Supabase -> Groq)**

**Speaker:**
"Our architecture is built for speed and scalability, completely decoupling the frontend from the backend. 

- **Frontend:** We use **Next.js** with modern styling to ensure a fluid, app-like user experience.
- **Backend:** Our core API is built on **FastAPI**, which is incredibly fast. When you chat with the bot, we use **Server-Sent Events (SSE)** to stream the AI's response token-by-token in real-time, completely eliminating loading screens.
- **RAG Engine (Retrieval-Augmented Generation):** To make the bot 'smart', we take your uploaded files, chunk the text, and convert them into vectors using **Nomic Embeddings**. We store these in a **Supabase PostgreSQL database** using the `pgvector` extension. This allows for sub-millisecond semantic search.
- **AI Brain:** When you ask a question, the system retrieves the most relevant chunks of knowledge and feeds them to our LLM—**Groq running Llama 3.3 70B**—for lightning-fast, accurate answers.
- **Storage:** We use **Cloudinary** for heavy document storage and **Supabase Buckets** for profile avatars."

## Part 3: Deep Dive into Asynchronous Workers

**(Visual: Diagram focusing on FastAPI passing tasks to RabbitMQ, which feeds into Celery Workers and Redis)**

**Speaker:**
"Now, processing large PDFs or running image recognition (OCR) takes a lot of computing power. If our main FastAPI server did this, it would freeze and other users wouldn't be able to chat. To solve this, we use an asynchronous worker architecture using three key cloud-based technologies:

1. **Celery Worker & Its Role:**
Celery is our asynchronous task queue. Its main role is to do the 'heavy lifting' in the background. For example, when you upload an image, the main server hands the image off to a Celery Worker. The worker runs the Tesseract OCR engine to extract the text, entirely separate from the main API. This keeps the user interface snappy and responsive.

2. **RabbitMQ (Cloud-based Message Broker):**
But how does the main server talk to Celery? Through RabbitMQ. RabbitMQ acts as a post office. When you upload a file, the FastAPI server drops a 'message' into RabbitMQ. RabbitMQ safely holds this message in a queue until a Celery worker is free to process it. By using a cloud-based instance, we ensure that no tasks are ever lost, even under heavy traffic.

3. **Redis (Cloud-based In-Memory Store):**
While RabbitMQ manages the line of tasks, we use Redis for extreme speed. Redis is an in-memory data structure store. It is typically used as a 'Result Backend'—meaning once Celery finishes extracting text from a massive PDF, it temporarily stores the result in Redis. The main API can instantly grab it from Redis without querying a slow hard drive. We also use it to cache active streaming states, ensuring our real-time chat never misses a beat.
